# Stemcell Idea Analysis: Multi-Agent "One Repo" Architecture

## Short answer
Yes — this is absolutely possible.

What you are describing is a **meta-repo (or mono-repo) of AI agents** where each upstream agent project is imported, trimmed, normalized, and exposed through one common invocation layer.

---

## What you want (restated)
You want one repository that can:

1. Import multiple upstream agent repos (e.g. `github-openclaw`, `github-gitclaw`, `github-minimum-intelligence`)
2. Automatically remove/cull unneeded files
3. Stay updated from each upstream `master/main`
4. Let users invoke any agent from one interface

This is feasible, and is a strong pattern for building a **curated agent distribution**.

---

## Recommended architecture

## 1) Keep upstream mirrors separate from runtime output
Use two layers:

- `upstream/` → clean synced copy of each source repo
- `agents/` → culled/normalized runnable version

Example:

```text
/
  upstream/
    github-openclaw/
    github-gitclaw/
    github-minimum-intelligence/

  agents/
    openclaw/
    gitclaw/
    minimum-intelligence/

  runtime/
    cli/
    adapters/
    contracts/

  manifests/
    agents.yaml
    cull-rules/
      openclaw.yaml
      gitclaw.yaml
      minimum-intelligence.yaml

  tools/
    sync-upstreams.sh
    cull-agents.sh
    validate-agents.sh
```

---

## 2) Upstream sync options

### Option A: Git submodules
- Pros: explicit link to source SHA, clean provenance
- Cons: friction for contributors, awkward automation UX

### Option B: Git subtree
- Pros: no extra submodule commands; easier contributor flow
- Cons: larger history; subtree maintenance complexity

### Option C: Snapshot/vendor import (scripted)
- Pros: simplest operationally, easiest to cull and rewrite
- Cons: you own full sync logic and provenance tracking

**Practical recommendation:** start with **scripted snapshot/vendor import** (Option C), then move to subtree if needed.

---

## 3) Invocation layer (single entrypoint)
Define a small contract each agent adapter must implement:

- `check()` → dependency/health check
- `run(task, context)` → execute agent
- `capabilities()` → what this agent supports

Then expose a unified command, e.g.:

```bash
stemcell run openclaw --task "review PR #123"
stemcell run gitclaw --task "summarize repo"
```

This avoids forcing all agents into the same internal code style.

---

## 4) Automated culling
Use declarative cull manifests per agent.

Example idea (`manifests/cull-rules/openclaw.yaml`):
- keep: `src/**`, `package.json`, `README.md`, `LICENSE*`
- remove: `docs/internal/**`, `examples/**`, `tests/e2e/**`, `.github/**`
- patch: rename env vars / config paths to avoid collisions

Pipeline:
1. Sync upstream
2. Apply cull rules
3. Apply normalization patches
4. Run smoke tests
5. Publish/update `agents/`

---

## 5) Auto-update from upstream master/main
Use GitHub Actions (scheduled + manual trigger):

1. Fetch latest upstream HEAD for each repo
2. Compare against last synced SHA
3. If changed:
   - import
   - cull
   - run validation/tests
   - open PR with changelog + SHAs

This gives predictable updates and reviewable diffs.

---

## Key risks and mitigations

1. **Dependency collisions**
   - Different Node/Python versions or package conflicts
   - Mitigation: per-agent isolated environments (container, venv, or per-agent workspace)

2. **License compatibility**
   - Not all upstream licenses can be mixed the same way
   - Mitigation: keep per-agent LICENSE + NOTICE, track provenance per file/import

3. **Security/supply chain risk**
   - Upstream updates may introduce malicious or breaking changes
   - Mitigation: pin SHAs, CI scans, mandatory PR review for sync bot PRs

4. **Breaking interface drift**
   - Upstream changes internal APIs unexpectedly
   - Mitigation: adapters + contract tests

5. **Repo size/performance**
   - Multiple agents can bloat one repo
   - Mitigation: cull aggressively, avoid committing artifacts, optionally split build artifacts

## Suggested implementation phases

### Phase 1 (MVP)
- Add one upstream (already appears you have minimum-intelligence locally)
- Build sync + cull scripts
- Add one adapter + single CLI command

### Phase 2
- Add openclaw + gitclaw
- Add per-agent contract tests
- Add scheduled auto-sync workflow

### Phase 3
- Add governance: agent registry, compatibility matrix, deprecation policy
- Add optional plugin system for future agent repos
