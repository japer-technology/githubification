# Lesson from Core

### What `japer-technology/github-core` teaches us about Type 2 — Non-AI Software Repo Githubification

---

## The Subject

[Home Assistant Core](https://github.com/home-assistant/core) is the most popular open-source home automation platform in the world — a Python 3 application that puts local control and privacy first, powered by a worldwide community of tinkerers and DIY enthusiasts. It is designed to run on a Raspberry Pi or a local server, connecting to thousands of IoT devices, sensors, switches, lights, climate systems, media players, and services. The platform exposes a web-based frontend, a REST API, a WebSocket API, and an automation engine that lets users define rules, scripts, and scenes for their smart homes. The project has over 2,000 integrations (called "components"), each living in its own subdirectory under `homeassistant/components/`, and is maintained by hundreds of code owners tracked in a 93KB auto-generated `CODEOWNERS` file. It requires Python 3.13+, uses an async-first architecture built on `asyncio`, and enforces strict typing, comprehensive linting (Ruff, PyLint), type checking (MyPy), and extensive testing (pytest). It is licensed under Apache 2.0 and is part of the Open Home Foundation.

`japer-technology/github-core` is the Githubification project. Unlike the native-agent case studies (GitClaw, GMI) or the agent-framework case studies (LangChain.js, Pydantic AI), this repository is **not an AI agent and not an AI framework**. It is a large, mature, non-AI software project — the kind of repository that makes up the overwhelming majority of GitHub. What the fork reveals is a repository that has already made itself **deeply AI-agent-accessible** without adding a Githubification folder: comprehensive `AGENTS.md` and `CLAUDE.md` files (identical, 42KB of structured coding guidelines for AI agents), a `.github/copilot-instructions.md` (auto-generated by `script/gen_copilot_instructions.py`), a `.claude/` directory with agent configurations and skills, a `.gemini/` directory with symlinked skills, an `.agent/` directory with symlinked skills, a `.devcontainer/` for reproducible development environments, and a full suite of GitHub Actions workflows for CI, security scanning, dependency management, issue triage, and build automation. The repository has been made **AI-agent-readable and AI-agent-developable** — but its core functionality (running a home automation server connected to physical devices) remains firmly outside what GitHub Actions can execute.

This is a Type 2 — Non-AI Software Repo Githubification. It presents the hardest and most common Githubification challenge: how do you Githubify software that was designed to run as a persistent daemon on local hardware, connected to physical devices that a GitHub Actions runner will never see?

---

## The Core Lesson

> **The largest category of repositories on GitHub — non-AI software that runs as long-lived services connected to external systems — cannot be fully executed on GitHub Actions. Githubification for these repositories is not about running the software; it is about making the software's development, testing, configuration, and knowledge accessible through GitHub-as-infrastructure. When a repository has already invested in AI agent readability (AGENTS.md, copilot instructions, agent skills) and GitHub Actions automation (CI, linting, security scanning, issue triage), the gap to Githubification is not technical — it is conceptual: adding an issue-driven AI agent that serves as a conversational interface to the project's capabilities, documentation, and development workflow.**

The previous case studies all share an assumption: the subject software can, in some form, execute on a GitHub Actions runner. GitClaw was born there. GMI runs natively. LangChain.js needs only Node.js. Pydantic AI needs only Python and `uv`. Even Agent Zero and AutoGPT, which can't run their full stacks on Actions, are AI agents whose core value (LLM interaction) can be delivered by a substitute agent running on the same runner.

Home Assistant Core breaks this assumption entirely. Its value proposition is not computation — it is **orchestration of physical devices**. A GitHub Actions runner cannot turn on your lights, read your thermostat, lock your front door, or play music through your speakers. The software's purpose is inseparable from hardware that exists in someone's home, not in a cloud data center. No amount of Githubification can change that.

This does not mean Githubification is impossible. It means the scope shifts. Instead of "run the software on GitHub," the question becomes: "What can an AI agent, running on GitHub Actions, do with this codebase that is useful?"

| GitHub Primitive | Current Use | Githubification Extension |
|---|---|---|
| **GitHub Actions** | Compute — CI/CD (lint, typecheck, test, build, wheels), CodeQL security scanning, duplicate issue detection, non-English issue detection, stale issue management, task creation restriction, translations | Would add issue-driven agent execution for development assistance, integration scaffolding, configuration validation, and documentation |
| **Git** | Storage — the entire codebase (2,000+ integrations), auto-generated files (CODEOWNERS, copilot instructions, requirements), Docker build artifacts | Would add session state, agent memory, generated configurations |
| **GitHub Issues** | UI — bug reports, feature requests, integration requests (structured templates) | Would become the conversational interface for interacting with the project's development capabilities |
| **GitHub Secrets** | Credential store — CI tokens, Docker registry credentials | Would add LLM API keys for the Githubification agent |

The four primitives map — but they map to the **development workflow**, not to the **runtime functionality**. This is the defining characteristic of Type 2 Githubification for infrastructure software.

## Anatomy of the Repository

The repository has six distinct layers relevant to Githubification:

### Layer 1: AI Agent Readability

The most striking feature of `github-core` from a Githubification perspective is how thoroughly it has been prepared for AI agents — not for running them, but for being developed by them:

```
AGENTS.md                              # 42KB — comprehensive coding guidelines for AI agents
CLAUDE.md                              # Identical to AGENTS.md (symlink or copy)
.github/copilot-instructions.md        # Auto-generated by script/gen_copilot_instructions.py
.agent/skills/                         # Symlink to shared skill definitions
.claude/agents/                        # Claude-specific agent configurations
.claude/skills/                        # Claude-specific skill packages
.gemini/skills/                        # Gemini-specific skills (symlink)
```

The `AGENTS.md` file is not a placeholder. It is a dense, structured document covering Python requirements (3.13+, pattern matching, type hints, f-strings, dataclasses, walrus operator), strict typing requirements, code quality standards (Ruff, PyLint, MyPy), writing style guidelines, documentation standards, async programming rules (executor usage, thread safety, callback decorators), error handling hierarchies (`ServiceValidationError`, `HomeAssistantError`, `ConfigEntryNotReady`, `ConfigEntryAuthFailed`, `ConfigEntryError`), logging conventions, development commands, and detailed anti-pattern/best-practice comparisons with code examples. This is not documentation for humans who happen to use AI — it is documentation **for AI agents** that happen to assist humans.

### Layer 2: GitHub Actions Workflows

```
.github/workflows/
├── builder.yml                        # Docker image builds
├── ci.yaml                            # 62KB — comprehensive CI pipeline (lint, type, test, build)
├── codeql.yml                         # Security scanning
├── detect-duplicate-issues.yml        # AI-powered duplicate issue detection
├── detect-non-english-issues.yml      # Non-English issue handling
├── lock.yml                           # Stale issue locking
├── restrict-task-creation.yml         # Task creation authorization
├── stale.yml                          # Stale issue/PR management
├── translations.yml                   # Translation synchronization
├── wheels.yml                         # Python wheel builds
└── matchers/                          # Custom problem matchers
```

The CI pipeline alone is 62KB — one of the most sophisticated GitHub Actions configurations in open source. It handles multi-version Python testing, integration-specific test sharding, coverage reporting, linting, type checking, dependency validation, and build verification. The `detect-duplicate-issues.yml` workflow is particularly notable: it uses AI to identify duplicate issues, demonstrating that AI-powered automation is already part of the repository's GitHub infrastructure.

### Layer 3: The Core Application

```
homeassistant/
├── __init__.py                        # Package marker
├── __main__.py                        # Entry point
├── core.py                            # 99KB — the HomeAssistant class, event bus, state machine
├── core_config.py                     # Configuration management
├── config.py                          # YAML configuration parsing
├── config_entries.py                  # 150KB — integration configuration entry system
├── bootstrap.py                       # Application startup sequence
├── runner.py                          # Event loop management
├── loader.py                          # Integration discovery and loading
├── setup.py                           # Integration setup orchestration
├── requirements.py                    # Dependency resolution
├── exceptions.py                      # Exception hierarchy
├── const.py                           # Constants and enumerations
├── auth/                              # Authentication system
├── components/                        # 2,000+ integrations
├── helpers/                           # Utility modules
├── util/                              # Low-level utilities
├── scripts/                           # Management scripts
├── generated/                         # Auto-generated code
├── brands/                            # Integration branding
└── backports/                         # Python version compatibility
```

The `components/` directory is the heart of the platform — each subdirectory is a self-contained integration with its own `manifest.json`, `__init__.py`, config flow, entity platforms, and tests. The modular architecture means each integration is independently understandable, independently testable, and independently developable. This modularity is what makes AI agent assistance practical despite the codebase's enormous size.

### Layer 4: Development Infrastructure

```
script/
├── gen_copilot_instructions.py        # Generates .github/copilot-instructions.md
├── hassfest/                          # Project structure validation
├── scaffold/                          # Integration scaffolding tools
├── translations/                      # Translation management
├── gen_requirements_all.py            # Requirements file generation
├── split_tests.py                     # Test sharding for CI
├── lint_and_test.py                   # Combined lint/test runner
├── version_bump.py                    # Version management
└── ...
```

The `script/hassfest/` tool validates that the project structure is correct — manifests are valid, CODEOWNERS is up to date, requirements are consistent, and generated files match their sources. The `script/scaffold/` tool generates new integrations from templates. These tools are critical for Githubification because they represent **codified project knowledge** — rules and patterns that an AI agent could enforce, explain, or apply through an issue-driven interface.

### Layer 5: Quality Enforcement

```
.core_files.yaml                       # Defines which files trigger full CI
CODEOWNERS                             # 93KB — auto-generated ownership mappings
mypy.ini                               # Type checking configuration
pyproject.toml                         # Project metadata, Ruff config, tool settings
.pre-commit-config.yaml                # Pre-commit hook configuration
.hadolint.yaml                         # Dockerfile linting
.yamllint                              # YAML linting
.strict-typing                         # Strict typing enforcement list
codecov.yml                            # Coverage configuration
pylint/                                # Custom PyLint plugins
```

The quality infrastructure is exceptionally thorough. `.core_files.yaml` defines a dependency graph that determines which test suites to run based on which files changed — core changes trigger the full suite, component changes trigger only relevant tests. This is a CI optimization pattern that most projects never implement but that a Githubification agent could leverage to provide targeted feedback.

### Layer 6: Containerization

```
Dockerfile                             # Production image (auto-generated by hassfest)
Dockerfile.dev                         # Development image
rootfs/                                # Container filesystem overlay
.devcontainer/devcontainer.json        # VS Code dev container configuration
```

The containerization layer provides reproducible environments for both production and development. The dev container is significant for Githubification: it proves that the development environment can be fully defined in the repository, which means a GitHub Actions runner (or Codespace) can recreate it.

## Key Patterns

### 1. AI Agent Instructions as Infrastructure

Home Assistant Core has invested more in AI agent readability than almost any other open-source project. The `AGENTS.md` file is not an afterthought — it is a 42KB structured document generated by a dedicated script (`script/gen_copilot_instructions.py`). It covers every aspect of contributing to the project: coding standards, error handling patterns, async programming rules, testing conventions, and common anti-patterns with corrected alternatives.

This is a pattern that applies to every Type 2 Githubification: **before you add an AI agent to a repository, make the repository legible to AI agents.** Home Assistant Core demonstrates what thorough agent readability looks like — not a paragraph of context, but a comprehensive instruction set that allows an AI agent to contribute with the same quality standards as a human maintainer.

### 2. Generated Configuration Files

The repository auto-generates several critical files:
- `CODEOWNERS` is generated by `script/hassfest/codeowners.py`
- `.github/copilot-instructions.md` is generated by `script/gen_copilot_instructions.py`
- `requirements_all.txt` is generated by `script/gen_requirements_all.py`
- The `Dockerfile` is generated by hassfest

This pattern — **generate, don't manually maintain** — is essential for large codebases. It ensures consistency and reduces drift. For Githubification, it also means that an AI agent can re-run these generators after making changes, validating its own work against the same rules that human contributors follow.

### 3. Modular Integration Architecture

Each of Home Assistant's 2,000+ integrations follows a standardized structure:

```
homeassistant/components/<integration>/
├── manifest.json          # Metadata: name, domain, requirements, dependencies, codeowners
├── __init__.py            # Setup and teardown
├── config_flow.py         # Configuration UI flow
├── const.py               # Integration constants
├── entity.py              # Base entity class
├── sensor.py              # Sensor platform (optional)
├── switch.py              # Switch platform (optional)
├── strings.json           # Translatable strings
└── ...
```

This modularity is the single most important architectural property for Githubification. An AI agent does not need to understand the entire 99KB `core.py` to help a user write a new integration. It needs to understand the integration contract — what a `manifest.json` must contain, what `async_setup_entry` must do, what entity platforms look like — and then apply that contract to a specific device or service. The scaffolding tool (`script/scaffold/`) already codifies this process. A Githubification agent could expose it through Issues: "Create an integration for my Acme thermostat" → the agent generates the scaffold, fills in the boilerplate, and commits.

### 4. Tiered CI Based on Change Scope

The `.core_files.yaml` defines categories of files and which test suites they trigger:

| Category | Scope | CI Response |
|---|---|---|
| `core` | `homeassistant/*.py`, `auth/`, `helpers/`, `util/` | Full test suite |
| `base_platforms` | 50+ platform components (sensor, switch, light, etc.) | Full test suite |
| `components` | 40+ infrastructure components (automation, MQTT, frontend, etc.) | Full test suite |
| `tests` | Test infrastructure, conftest files, test utilities | Full test suite |
| `requirements` | Requirements files, constraints, pyproject.toml | Dependency validation |

This tiered approach means the CI system already knows what's important. A Githubification agent could use this same mapping to scope its analysis — when a user asks about a specific integration, the agent knows which files matter and which tests to reference.

### 5. Issue Triage Automation

The repository already uses GitHub Actions for AI-powered issue management:
- **Duplicate detection** (`detect-duplicate-issues.yml`) — AI identifies potential duplicate issues
- **Non-English detection** (`detect-non-english-issues.yml`) — handles non-English submissions
- **Stale management** (`stale.yml`) — auto-closes inactive issues
- **Task restriction** (`restrict-task-creation.yml`) — limits who can create certain issue types
- **Lock** (`lock.yml`) — locks resolved conversations

These workflows prove that GitHub Actions is already the execution environment for AI-powered automation on this repository. Githubification would extend this pattern from **reactive triage** to **proactive assistance**.

### 6. Multi-Agent Readability

The repository doesn't just support one AI agent — it supports at least four:
- **GitHub Copilot** via `.github/copilot-instructions.md`
- **Claude Code** via `CLAUDE.md` and `.claude/`
- **Gemini** via `.gemini/`
- **Generic agents** via `AGENTS.md` and `.agent/`

This multi-agent readability is significant because it means the repository has already solved the "agent instructions" problem at scale. Any Githubification agent added to this repository would inherit a rich context about coding standards, architectural patterns, and development workflows without needing to build that context from scratch.

### 7. The Hardware Boundary

Home Assistant Core's fundamental purpose — controlling physical devices in someone's home — creates an absolute boundary that no Githubification strategy can cross. A GitHub Actions runner cannot:

- Discover devices on a local network (Zigbee, Z-Wave, Wi-Fi, Bluetooth)
- Read sensor data (temperature, humidity, motion, power)
- Control actuators (lights, switches, locks, thermostats)
- Run persistent automations (sunrise triggers, presence detection)
- Serve a web dashboard to local browsers

This boundary is not a limitation of the Githubification approach — it is a **classification tool**. It tells us precisely what Type 2 Githubification is for: not replacing the software's runtime, but augmenting the software's development, configuration, and knowledge management through an AI agent on GitHub.

## What Core Teaches That Agent Repos Don't

The Type 1 case studies (GitClaw, GMI, OpenClaw, MicroClaw) demonstrate how to Githubify AI agents — software whose core value (LLM interaction) can execute on a GitHub Actions runner. The framework case studies (LangChain.js, Pydantic AI) demonstrate how to Githubify agent-building tools — software whose development workflow maps natively to GitHub Actions. Home Assistant Core demonstrates a third category: software whose runtime is fundamentally incompatible with GitHub Actions but whose development ecosystem is already deeply integrated with GitHub.

1. **The biggest repositories are Type 2.** Most code on GitHub is not AI agents or AI frameworks. It is web applications, mobile backends, IoT platforms, game engines, operating system kernels, database engines, and infrastructure tools. These are all Type 2 candidates. Home Assistant Core, with its 2,000+ integrations and hundreds of maintainers, represents the scale and complexity that Type 2 Githubification must handle.

2. **AI agent readability precedes Githubification.** Home Assistant Core invested in `AGENTS.md`, copilot instructions, and multi-agent skill configurations before any Githubification layer was considered. This is not accidental — it is the natural first step. Before an AI agent can operate through Issues, it must understand the codebase. Home Assistant's investment in agent readability is the foundation on which Githubification would build.

3. **Development assistance is the primary value.** For a Type 2 repo with a hardware boundary, the Githubification agent's value is in the development lifecycle: scaffolding new integrations, validating configurations, explaining architectural patterns, reviewing code against the project's standards, generating tests, updating documentation, and triaging issues. These are high-value activities that an AI agent can perform entirely within GitHub Actions.

4. **Existing CI/CD is a head start.** The 62KB `ci.yaml` and ten additional workflows represent thousands of hours of automation investment. A Githubification agent doesn't need to recreate this — it needs to leverage it. When a user asks "Why is my integration failing?", the agent can read CI logs, identify the failing test, explain the error, and suggest a fix — all through the existing GitHub Actions infrastructure.

5. **Modularity enables scope.** A 99KB `core.py` is impenetrable to an AI agent in a single context window. But a single integration — 5 to 15 files, each following a standard pattern — is perfectly tractable. Home Assistant's modular architecture means a Githubification agent can operate at the integration level, providing focused, high-quality assistance without needing to comprehend the entire system.

## The Githubification Strategy for Type 2 Infrastructure Software

Home Assistant Core suggests a specific Githubification approach for non-AI software with a hardware boundary:

### Strategy: Development Agent

Instead of wrapping or substituting the software's runtime, insert an AI agent that serves the software's **development community** through Issues:

| Capability | How It Works on GitHub |
|---|---|
| **Integration scaffolding** | User opens an issue: "Create a sensor integration for Acme API." Agent runs `script/scaffold/`, fills in boilerplate, commits the result as a PR. |
| **Configuration validation** | User pastes a YAML configuration. Agent validates it against the integration's schema and reports errors or suggestions. |
| **Code review assistance** | Agent reviews PRs against the 42KB `AGENTS.md` coding standards, catching anti-patterns before human reviewers. |
| **Test generation** | User describes a scenario. Agent generates pytest tests following the project's fixture and mock conventions. |
| **Documentation** | Agent answers questions about architectural patterns, integration contracts, and async programming rules using the codebase as context. |
| **Issue triage** | Agent categorizes, deduplicates, and provides initial analysis of bug reports — extending the existing AI-powered triage workflows. |
| **Dependency analysis** | Agent analyzes `manifest.json` requirements, checks for version conflicts, and validates against `package_constraints.txt`. |

This agent would live in a `.github-core/` folder, following the same structural patterns as every other Githubified repo:

```
.github-core/
├── AGENTS.md                          # Agent identity and standing orders
├── github-core-ENABLED.md             # Sentinel file
├── lifecycle/
│   ├── github-core-ENABLED.ts         # Fail-closed guard
│   ├── github-core-INDICATOR.ts       # 👀 reaction
│   └── github-core-AGENT.ts           # Core orchestrator
├── install/
│   ├── github-core-INSTALLER.ts       # Bootstrap script
│   └── github-core-WORKFLOW-AGENT.yml # GitHub Actions workflow
├── state/
│   ├── issues/                        # Issue-to-session mappings
│   └── sessions/                      # Conversation transcripts
└── package.json                       # Runtime dependency
```

The agent would inherit the repository's existing `AGENTS.md` as its understanding of coding standards, use the `script/hassfest/` tool for validation, reference the `script/scaffold/` tool for integration generation, and read the CI workflow outputs for test and lint analysis.

## Lessons for Any Type 2 Githubification

1. **Start with agent readability.** Before adding a Githubification folder, make the repository legible to AI agents. Write `AGENTS.md`. Generate copilot instructions. Document coding standards with examples and anti-patterns. Home Assistant Core's 42KB instruction set is the gold standard.

2. **Identify the hardware boundary.** Every Type 2 repo has something it does that GitHub Actions cannot. Name it explicitly. For Home Assistant, it is physical device control. For a web application, it might be database access or user authentication. For a game engine, it might be GPU rendering. The boundary defines what the Githubification agent can and cannot do.

3. **Focus on the development lifecycle.** Inside the hardware boundary, the agent cannot help. Outside it — scaffolding, validation, code review, testing, documentation, triage — the agent can provide enormous value. Design the Githubification agent's capabilities around the development workflow, not the runtime functionality.

4. **Leverage existing CI/CD.** A mature repository's CI pipeline is a rich source of automated feedback. The Githubification agent should read CI outputs, interpret test failures, explain lint errors, and suggest fixes. Do not duplicate what CI already does — extend it with conversational context.

5. **Exploit modularity.** Large codebases are tractable when they are modular. Home Assistant's integration architecture means each conversation can be scoped to a single component. If the codebase is monolithic, the Githubification agent's first task should be helping users navigate it — a code-aware documentation assistant.

6. **Reuse existing tools.** Home Assistant has `hassfest` for validation, `scaffold` for generation, `gen_copilot_instructions.py` for documentation. A Githubification agent should call these tools, not reinvent them. The agent's job is to be the conversational interface to tools that already exist.

7. **Multi-agent readiness is free.** If the repository already supports Copilot, Claude, and Gemini through separate instruction files, adding a Githubification agent is a natural extension. The agent readability investment applies to all AI agents equally — including the one that will respond to Issues.

8. **Scale through integration boundaries.** A 2,000-integration codebase is too large for any single context window. But a Githubification agent that scopes each conversation to a single integration — identified by its `manifest.json` domain — can provide expert-level assistance within that scope. The architecture of the software defines the architecture of the agent's knowledge.

9. **Document the strategy explicitly.** Users and contributors need to understand what the Githubification agent can and cannot do. A Type 2 agent that promises to "run Home Assistant on GitHub" will fail and disappoint. A Type 2 agent that promises to "help you develop, test, and debug Home Assistant integrations through Issues" sets correct expectations and delivers real value.

10. **The agent augments, not replaces.** Type 1 Githubification replaces local execution with GitHub execution — the agent IS the runtime. Type 2 Githubification augments local execution with GitHub-based AI assistance — the agent is a development companion. Home Assistant Core will always need to run on local hardware to control devices. The Githubification agent makes building, testing, and maintaining that local installation easier.

## Summary

`japer-technology/github-core` demonstrates that the most common Githubification challenge is not "how do I run this AI agent on GitHub?" but "how do I make this non-AI software accessible through GitHub-as-infrastructure?" Home Assistant Core — the world's most popular open-source home automation platform — is a massive Python application designed to run as a persistent daemon on local hardware, connected to thousands of physical IoT devices. Its core functionality cannot execute on a GitHub Actions runner. No amount of Githubification can make a CI machine turn on your lights.

But the repository has already completed the most important preparatory step: it has made itself deeply AI-agent-readable. A 42KB `AGENTS.md`, auto-generated copilot instructions, multi-agent skill configurations for Claude, Gemini, and generic agents, a modular integration architecture with standardized contracts, codified development tools (hassfest, scaffold, generators), and a sophisticated CI pipeline that already includes AI-powered issue triage. The foundation for Githubification is built.

The lesson for Type 2 Githubification is that the agent's value is in the **development lifecycle**, not the **runtime functionality**. Integration scaffolding, configuration validation, code review, test generation, documentation, issue triage, and dependency analysis — these are all high-value capabilities that an AI agent can deliver through Issues on GitHub Actions, using the codebase as context and the existing development tools as capabilities. The hardware boundary is not a failure of Githubification — it is the boundary that defines the agent's scope.

Where the Type 1 case studies prove that AI agents can run on GitHub, and the framework case studies prove that AI tools can be composed on GitHub, Home Assistant Core proves that **any repository** — regardless of size, complexity, or runtime requirements — can benefit from a Githubification agent. The agent just needs to know its role: not to replace the software, but to make the software easier to build, test, maintain, and understand.
